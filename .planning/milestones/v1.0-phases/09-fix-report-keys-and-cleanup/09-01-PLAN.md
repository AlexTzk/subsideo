**BUG-2:** Change `_CRITERIA_MAP["phase_rms_rad"]` from `("phase RMS < 0.05 rad", "phase_rms_lt_0.05")` to `("phase RMS < 0.05 rad", "phase_rms_lt_0.05rad")`.

**Correlation ambiguity:** The `correlation` field exists in both `RTCValidationResult` (where the pass_criteria key is `correlation_gt_0.99`) and `DISPValidationResult` (where the pass_criteria key is `correlation_gt_0.92`). The static `_CRITERIA_MAP` cannot distinguish these.

Fix: Instead of relying solely on `_CRITERIA_MAP` for the criterion key, update `_metrics_table_from_result()` to also do a reverse lookup — if the `_CRITERIA_MAP` criterion_key is not found in `pass_criteria`, scan `pass_criteria` for any key starting with the field name (e.g., `correlation`). This handles the DISP case where `correlation` maps to `correlation_gt_0.92` rather than `correlation_gt_0.99`.

Concrete change to `_metrics_table_from_result()`: after line `passed = pass_criteria.get(criterion_key) if criterion_key else None`, add a fallback:

```python
# Fallback: if static map key not found, search pass_criteria for
# a key starting with the field name (handles correlation ambiguity
# between RTC correlation_gt_0.99 and DISP correlation_gt_0.92)
if passed is None and criterion_key:
    for pc_key, pc_val in pass_criteria.items():
        if pc_key.startswith(f.name):
            passed = pc_val
            criterion_label = pc_key  # show actual criterion
            break
```

Also update the `_CRITERIA_MAP["correlation"]` entry to use a more generic label: `("r > threshold", "correlation_gt_0.99")` — the fallback will override it for DISP.
