### 4.1 Single-event F1 variance + block-bootstrap CI methodology

For binary-classification metrics on a single 100km MGRS tile, the i.i.d.
assumption underlying naive standard errors is violated: nearby pixels
share land-cover, atmospheric correction, and orbit-pass conditions, so
the effective sample size is far smaller than the pixel count. Hall (1985)
introduced the spatial block bootstrap to address this; Lahiri (2003)
"Resampling Methods for Dependent Data" Ch.7-8 is the canonical reference.

**Implementation** (`src/subsideo/validation/bootstrap.py` from Plan 05-03):

- **Block size = 1000 m** (`DEFAULT_BLOCK_SIZE_M`). At 30 m posting this
  is 33×33 pixels per block; on a 3660×3660 MGRS tile, that yields
  110×110 = 12,100 full blocks. Partial blocks at east + south edges
  (the L-shaped strip) are dropped per CONTEXT D-08; the dropped count
  for a 100km tile is (110+1)² − 110² = 221 partial blocks (~1.6% of
  pixel count). `n_blocks_kept` and `n_blocks_dropped` are reported in
  `metrics.json` per_event[N].bootstrap_config for transparency.

- **B = 500 resamples** (`DEFAULT_N_BOOTSTRAP`). Each resample draws
  `n_blocks_kept` block indices with replacement; the metric (F1 /
  precision / recall / accuracy) is computed over the union of resampled
  blocks' pixels. The 95% CI is the [2.5, 97.5] percentile of the
  bootstrap distribution. CI bounds are NOT assumed symmetric (binary
  classification F1 near boundary can produce skew).

- **PCG64 fixed seed = 0** (`DEFAULT_RNG_SEED`). `numpy.random.default_rng(0)`
  is the modern NumPy-recommended generator (per SciPy SPEC 7); Mersenne
  Twister `RandomState` is rejected as legacy + slower. The seed is
  serialised in `metrics.json bootstrap_config.rng_seed` for reproducibility
  audit.

**Runtime:** ~3 minutes per metric per 100km tile at B=500 (verified per
RESEARCH Probe 9). For a 3-event Phase 5 aggregate × 4 metrics = 12
bootstrap calls = ~36 minutes; well within the 8-hour `EXPECTED_WALL_S`
budget. Optimisation (numba / vectorised gather) is deferred until a
2nd consumer appears.

**Switching defaults:** changing `DEFAULT_BLOCK_SIZE_M` / `_N_BOOTSTRAP` /
`_RNG_SEED` requires a visible PR diff in `bootstrap.py`; auditable via
`git log --grep='DEFAULT_BLOCK_SIZE_M'`. Never an env var or CLI flag.
This is the same posture as the Phase 4 §3.4 `REFERENCE_MULTILOOK_METHOD`
constant.

### 4.2 EFFIS rasterisation choice — `all_touched=False` as gate value

EFFIS publishes burnt-area perimeters as vector polygons (GML or
shapezip via WFS); subsideo's DIST-S1 reference grid is a 30 m raster
(matching OPERA spec). Rasterising the polygons onto the raster requires
choosing whether boundary-touching pixels count as "burnt"
(`all_touched=True`) or only pixels whose centre falls inside the polygon
(`all_touched=False`).

**Phase 5 default: `all_touched=False`** (CONTEXT D-17). Rationale:

- **Conservative** — does not inflate recall via boundary-only labelling.
  PITFALLS P4.4 documents a 2-4 percentage-point F1 swing between the two
  modes for typical Sentinel-1 burn scars where the boundary perimeter is
  a substantial fraction of the polygon area.

- **Reproducible** — `rasterio.features.rasterize` default is
  `all_touched=False`; explicitly passing the literal in
  `validation/effis.py::rasterise_perimeters_to_grid()` makes intent
  grep-able (`grep -c "all_touched=False" src/subsideo/validation/effis.py`
  outputs at least 1).

**Diagnostic delta** (CONTEXT D-17): the eval script ALSO computes
`all_touched=True` F1 as a narrative-only diagnostic. Each event's
`metrics.json per_event[N].rasterisation_diagnostic` records:

```json
{
  "all_touched_false_f1": 0.823,   // gate value
  "all_touched_true_f1":  0.857,   // diagnostic
  "delta_f1": 0.034                 // expected ~+0.02..+0.04
}
```

The delta surfaces the structural F1 inflation from boundary-only
labelling without changing the gate. CONCLUSIONS_DIST_EU.md per-event
narrative reports both numbers + the delta.

### 4.3 EFFIS class-definition mismatch caveat

EFFIS's "burnt area" class definition does not match DIST-S1's "disturbed"
class definition. PITFALLS P4.5 documents the bidirectional mismatch:

- **EFFIS includes but DIST may miss:**
  - Low-severity ground fire (sparse vegetation that DIST's backscatter
    threshold may not register as a change)
  - Burn-and-resprout scars (where post-fire greening within weeks
    obscures the backscatter signal)

- **DIST includes but EFFIS does NOT cover:**
  - Clear-cut logging in unburnt forest (the original Romania 2022 use
    case — substituted with Spain Culebra fire per RESEARCH Probe 4
    because EFFIS is fire-only)
  - Windthrow / blowdown events
  - Defoliation from pest outbreaks
  - Construction / land-use change

**Implication for Phase 5 metrics:** precision and recall against EFFIS
are bounded by this class mismatch. The DIST-05 thresholds (precision >
0.70 AND recall > 0.50) are calibrated for this bound — they do NOT
require precision = recall = 1.0 because the references disagree on
class definitions. A FAIL on these thresholds is a real subsideo
deficiency; a borderline PASS (e.g., precision = 0.72, recall = 0.55) is
a methodologically-bound result, not a tight PASS.

This caveat is narrative-only; the gate values are not adjusted. Future
work (DIST-V2-04 or beyond) may layer in a windthrow / clear-cut reference
(Global Forest Cover Change products) for the inverse-direction class.
