    ## 1. CSLC cross-version phase impossibility

    <a name="cross-version-phase"></a>

    **TL;DR:** Interferometric phase comparison between CSLCs produced by
    different isce3 **major** versions yields coherence ≈ 0 regardless of the
    phase corrections applied on top. Amplitude-based metrics remain valid
    across versions. **Do NOT re-attempt with additional corrections.**

    ### 1.1 Structural argument — SLC interpolation kernel changed

    Between isce3 0.15 and 0.25 (and likely at or before isce3 0.19 — see
    upstream [isce3 Releases](https://github.com/isce-framework/isce3/releases)),
    the **SLC interpolation kernel itself** changed. This is upstream of any
    phase-screen correction:

    1. The SLC geocoding step samples the native-grid Sentinel-1 SLC complex
       values onto the target geocoded grid. Sub-pixel sample weights are set
       by the interpolation kernel.
    2. Changing the kernel (e.g. sinc → raised-cosine, kernel-width change,
       sub-kernel optimisation) changes the complex value at every geocoded
       pixel by a sub-pixel-scale-phase amount.
    3. **Two CSLCs computed from the same SLC with different interpolation
       kernels produce different absolute phase values at every pixel.** No
       amount of downstream phase correction (carrier removal, flattening,
       tropospheric correction, solid Earth tide, ionospheric delay, range
       bistatic correction, azimuth FM rate variation) can recover the
       original phase values.

    The only "correction" that would recover cross-version phase equivalence
    is **regenerating the OPERA CSLC with the newer isce3 kernel**, at which
    point it is just re-running the pipeline and the comparison is no longer
    a cross-version check.

    ### 1.2 Policy statement

    **Do NOT re-attempt with additional corrections.** A PR titled "CSLC phase
    comparison: additional corrections" touching
    `src/subsideo/validation/compare_cslc.py` or `src/subsideo/validation/compare_disp.py`
    **MUST address why the SLC-interpolation-kernel argument no longer holds**
    before merge. Adding another correction term (e.g. ionospheric, solid
    Earth tide, fallback flattening) does NOT constitute addressing the
    argument — those are all downstream of the kernel. The only merge-able
    argument is empirical evidence of kernel stability across the specific
    isce3 version pair at issue (e.g. 0.25.8 → 0.25.10 within the same minor
    series — see PITFALLS.md §P2.4 for the optional diagnostic-experiment
    menu, LOW-confidence validation).

    Reviewers: a rerun showing coherence 0.0003 → 0.0015 is NOT progress;
    both are random noise. Any "subtract X and Y and Z" PR description is a
    code smell.

    ### 1.3 Diagnostic Evidence (Appendix)

    The following is *corroborating evidence*, not the lead argument. v1.0
    diagnostic experiments on burst `t144_308029_iw1` comparing our chain
    (isce3 0.25.8) against OPERA reference (isce3 0.15.1) produced:

    | Correction removed from both | Coherence |
    |------------------------------|-----------|
    | None (raw compare)           | 0.0003    |
    | Carrier phase                | 0.002     |
    | Flattening phase             | 0.003     |
    | Carrier + flattening         | 0.002     |
    | OPERA-reference carrier+flatten applied to our product | 0.002 |

    All values are random-noise coherence. The phase difference has std
    2.57 rad (carrier) and 2.59 rad (flattening) — uniform-over-2π,
    uninformative. See `CONCLUSIONS_CSLC_N_AM.md` §5.3 for the full v1.0
    diagnostic report.

    **Implication:** amplitude-based cross-version validation (correlation +
    RMSE on magnitude) is the correct reference-agreement path when isce3
    majors differ. `subsideo.validation.compare_cslc.compare_cslc` returns
    amplitude-based `ReferenceAgreementResult`; the BINDING criteria
    `cslc.amplitude_r_min = 0.6` and `cslc.amplitude_rmse_db_max = 4.0` live
    in `subsideo.validation.criteria`.

    ### 1.4 Acceptable cross-version validation strategies

    1. **Amplitude-based single-scene comparison** — the standard v1.0+ path.
    2. **Self-consistency validation** on our own chain — Phase 3 CSLC
       self-consistency + Phase 4 DISP self-consistency use this strategy:
       14 sequential 12-day IFGs internal to our chain are coherence-valid
       even if they would not be coherent with an OPERA CSLC from a
       different isce3 version.
    3. **Time-series scientific equivalence** — compare displacement
       velocity from our CSLCs vs OPERA's across the same epoch pair (Phase
       4 DISP cross-chain scope, NOT Phase 3 single-chain scope).

    Phase-coherent single-scene cross-version comparison is **not on the
    acceptable-strategies list** regardless of how many corrections are
    layered. See §1.2.

    ## 2. Product-quality vs reference-agreement distinction

    **TL;DR:** Every v1.1 matrix cell reports **two distinct categories of
    measurement** that must not be conflated: product-quality gates assess
    whether *our* product is internally consistent; reference-agreement gates
    assess whether *our* product matches an *external* reference. The
    categories use different criteria and move based on different evidence.

    ### 2.1 Definitions

    | Category | Measures | Examples | Tightening criterion |
    |----------|----------|----------|----------------------|
    | **Product-quality** | Internal consistency of our chain against a target physical model | self-consistency coherence (Phase 3 CSLC, Phase 4 DISP), residual velocity on stable terrain, DSWx F1 against JRC (the JRC "reference" is actually a product-quality label source — see §4 for precision-first framing) | Evidence that our chain can consistently deliver a tighter bar (typically stack-length-dependent for InSAR self-consistency — see CSLC-V2-02 deferral) |
    | **Reference-agreement** | Agreement between our chain and an external reference product (OPERA, EGMS, JRC for water-body extent) | OPERA CSLC amplitude correlation/RMSE, OPERA DISP correlation, EGMS L2a bias | Never. See §2.3. |

    Dataclasses in `subsideo.validation.results` enforce the distinction:
    `ProductQualityResult` and `ReferenceAgreementResult` are structurally
    separate (no `.passed` collapse); `results/matrix.md` renders them in
    structurally separate columns.

    ### 2.2 Motivating example — Iberian Meseta CSLC-S1 three-number row

    `CONCLUSIONS_CSLC_EU.md` §6 reports **three independent numbers** for
    the Iberian Meseta burst, none of which collapses into the others:

    | Number | Category | Measurement | Criterion | Phase-3 verdict |
    |--------|----------|-------------|-----------|-----------------|
    | (a)    | reference-agreement | OPERA CSLC amplitude `r` + `RMSE_dB` | `cslc.amplitude_r_min=0.6`, `cslc.amplitude_rmse_db_max=4.0` (BINDING) | PASS/FAIL reportable |
    | (b)    | product-quality | self-consistency coherence `median_of_persistent` | `cslc.selfconsistency.coherence_min=0.7` (CALIBRATING) | `calibration data point 3` |
    | (c)    | product-quality | self-consistency residual `residual_mm_yr` + EGMS L2a stable-PS residual `egms_l2a_stable_ps_residual_mm_yr` | `cslc.selfconsistency.residual_mm_yr_max=5.0` (CALIBRATING) | `calibration data point 3` |

    If (a) PASSes but (b) is low → the chain is geocoding correctly but has
    an internal phase-temporal-stability problem. The honest response is NOT
    "relax (b) because (a) PASSes" — that's M4 goalpost-moving.

    If (a) is marginal (e.g. r = 0.62) but (b) is strong (coh = 0.85) → the
    chain is self-consistent but geocodes differently from OPERA. The honest
    response is NOT "tighten (a) to 0.8 now that we're so close" — that's
    M1 target-creep.

    ### 2.3 Target-creep prevention (M1)

    Reference-agreement thresholds **must not tighten** based on the
    reference's own score, even when per-cell reference-agreement
    dramatically outperforms the criterion:

    | Cell | Criterion | Measured | Anti-pattern response | Correct response |
    |------|-----------|----------|-----------------------|------------------|
    | RTC N.Am. | RMSE < 0.5 dB | 0.045 dB | Tighten to < 0.1 dB | Leave at < 0.5 dB. The 10× headroom is the reference's own quality, not a commitment we should inherit. |
    | EU RTC (per burst) | RMSE < 0.5 dB | 0.04–0.1 dB | Tighten to < 0.15 dB | Leave at < 0.5 dB. See RTC-02 explicit anti-tighten clause. |
    | CSLC amplitude | r > 0.6 | 0.79 | Tighten to r > 0.75 | Leave at r > 0.6. Tightening locks us to OPERA's interpolation kernel bias as the ceiling (see §1.1). |

    `subsideo.validation.criteria` enforces this in code: every criterion is
    a `@dataclass(frozen=True)` (tightening requires a PR diff citing an ADR
    or upstream spec change) plus `matrix_writer` echoes the criterion
    threshold alongside the measured value so drift is visible.

    ### 2.4 Goalpost-moving prevention (M4)

    Product-quality thresholds **must not relax** based on the measured
    value:

    | Cell | Criterion | Measured | Anti-pattern response | Correct response |
    |------|-----------|----------|-----------------------|------------------|
    | DSWx EU (any AOI) | F1 > 0.90 | 0.87 | Relax to F1 > 0.85 | Leave at F1 > 0.90. Report "FAIL with named ML-replacement upgrade path" per DSWX-07 discipline. |
    | CSLC self-consistency (SoCal/Mojave/Iberian) | coh > 0.7 | 0.68 | Relax to coh > 0.6 | CALIBRATING discipline — report the measurement, not a verdict. The gate METRIC may change (D-04 escape valve) but the VALUE stays. |

    - CSLC self-consistency: SoCal / Mojave / Iberian = 3 data points (this

    - DISP self-consistency: SoCal / Bologna = 2 data points (Phase 4); third

    CALIBRATING cells render italicised in `results/matrix.md` per Phase 1
    D-03; metadata field `binding_after_milestone` on the `Criterion`
