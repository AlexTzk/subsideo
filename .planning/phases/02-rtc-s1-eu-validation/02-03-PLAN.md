<objective>
Extend `matrix_writer.py` with two coordinated changes that make the Phase 2 `rtc:eu` cell render correctly AND prevent INVESTIGATION_TRIGGER criteria from polluting any cell's PQ/RA columns:

1. **Schema detection branch:** when `metrics.json` contains a `per_burst` key (D-11 marker for the RTCEUCellMetrics shape), render the cell as `X/N PASS` (or `X/N PASS ⚠` when `any_investigation_required`) instead of the default single-row PQ + RA rendering. PQ column reads "—" for RTC (no product-quality gate in v1.1).

2. **INVESTIGATION_TRIGGER filter in `_render_cell_column`:** skip any criterion_id whose `CRITERIA[cid].type == "INVESTIGATION_TRIGGER"` (defence in depth — the eval script should never put these in a `ReferenceAgreementResultJson.criterion_ids` list, but if it does, the matrix writer must not render a spurious `value (>= 0.15 PASS/FAIL)` verdict).

Purpose: Without this plan, Plan 04's metrics.json either renders as a confusing dump of per_burst fields (because the existing writer passes through MetricsJson.model_validate_json) or fails validation because RTCEUCellMetrics adds required fields (pass_count/total/all_pass/any_investigation_required) that the base MetricsJson doesn't have.

Output: matrix_writer.py has one new helper `_render_rtc_eu_cell(metrics: RTCEUCellMetrics) -> tuple[str, str]`; the `write_matrix` loop calls it when the raw JSON has a `per_burst` key; `_render_cell_column` filters INVESTIGATION_TRIGGER criteria. Tests cover both branches.

<note>
**⚠ Glyph convention (Plan 02-03 / 02-04 cross-file reference).**

Two Python source files in Phase 2 reference the warning glyph, but they do so in syntactically different forms that render identically as ⚠ at read time:

- `matrix_writer.py` uses the `⚠` unicode escape form. Rationale: byte-level repeatable across editors, terminals, and git diff; deterministic for Plan 02-03 tests that grep for `u26a0` in the source. The string-literal source bytes are ASCII-only.
- `run_eval_rtc_eu.py` (Plan 02-04) uses the literal ⚠ glyph inline in string bodies. Rationale: Python's AST parser reads utf-8 source natively, and the eval script's `print()` summary banners benefit from the glyph being directly visible during a log tail.

Both forms compile to the same `str` object and render as the same character when the matrix or summary is displayed. The convention is: matrix_writer stays escape-form for deterministic source bytes; eval scripts may use inline glyphs where useful for readability. No runtime difference.
</note>

    ```python
    def _render_cell_column(
        result: ProductQualityResult | ReferenceAgreementResult | None,
    ) -> str:
        """Render one side of a cell (product-quality or reference-agreement column).

        Returns an em-dash when there are no criteria to evaluate (empty side,
        matching CONTEXT.md §Claude's Discretion -- no-gate cells show '—').
        Cells with ANY CALIBRATING criterion are italicised as a whole (GATE-03).

        INVESTIGATION_TRIGGER criteria (Phase 2 D-13) are silently filtered out:
        they are non-gate markers and must NOT render a PASS/FAIL verdict in

        them to a ReferenceAgreementResult.criterion_ids list, but this filter
        provides defence-in-depth (an accidental inclusion renders as if the
        criterion wasn't there, rather than producing a misleading verdict).
    ```

    ```python
    def _is_rtc_eu_shape(metrics_path: Path) -> bool:
        """Return True if the metrics.json at ``metrics_path`` has a top-level ``per_burst`` key.

        Cheap schema-discrimination check (D-11 marker). Called only after
        ``_load_metrics`` has already confirmed the file exists and parses as
        valid JSON via MetricsJson. Returns False on any I/O or JSON error --
        the caller falls through to the default cell-render path, which then
        surfaces RUN_FAILED if ``_load_metrics`` also failed.
        """
        import json as _json

        try:
            raw = _json.loads(metrics_path.read_text())
        except (OSError, ValueError) as e:
            logger.debug("_is_rtc_eu_shape: cannot read {}: {}", metrics_path, e)
            return False
        return isinstance(raw, dict) and "per_burst" in raw


    def _render_rtc_eu_cell(
        metrics_path: Path,
    ) -> tuple[str, str] | None:
        """Render RTC-EU multi-burst aggregate as (pq_col, ra_col).

        Returns None if the metrics.json cannot be parsed as RTCEUCellMetrics
        (the caller should fall through to RUN_FAILED rendering). Returns a
        tuple of pre-escaped column strings when successful.

        pq_col is always the em-dash literal "—" because RTC has no
        product-quality gate in v1.1 (Phase 1 D-04; Phase 2 §specifics
        "No product-quality gate for RTC in v1.1").

        ra_col format: "{pass_count}/{total} PASS" (or "... PASS ⚠" when
        any_investigation_required is True, per D-15). When pass_count <
        total, format is "X/N PASS (Y FAIL)" so the matrix reader sees at
        a glance there are failing rows without having to open the CONCLUSIONS
        document.
        """
        from subsideo.validation.matrix_schema import RTCEUCellMetrics

        try:
            metrics = RTCEUCellMetrics.model_validate_json(metrics_path.read_text())
        except Exception as e:  # pydantic.ValidationError or JSON error
            logger.warning(
                "Failed to parse RTCEUCellMetrics from {}: {}", metrics_path, e
            )
            return None

        fail_count = metrics.total - metrics.pass_count
        if fail_count > 0:
            base = f"{metrics.pass_count}/{metrics.total} PASS ({fail_count} FAIL)"
        else:
            base = f"{metrics.pass_count}/{metrics.total} PASS"

        warn = " ⚠" if metrics.any_investigation_required else ""
        ra_col = f"{base}{warn}"
        pq_col = "—"
        return pq_col, ra_col
    ```
