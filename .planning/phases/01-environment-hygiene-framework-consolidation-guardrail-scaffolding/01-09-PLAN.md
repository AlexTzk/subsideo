**Env-name note (conda-env.yml `name: subsideo` vs Docker `-n base`):** The conda-env.yml authored in Plan 01-01 sets `name: subsideo` — that applies to host env creation via `micromamba env create -f conda-env.yml` (creates the `subsideo` env on the user's machine). Inside the Dockerfile, `micromamba install -y -n base -f /tmp/conda-env.yml` installs the same package set into the BASE env, because the `mambaorg/micromamba` base image pre-defines `MAMBA_ROOT_PREFIX=/opt/conda` + a default `base` env that's auto-activated via `_entrypoint.sh`. Overriding `-n base` avoids a second redundant env inside the container. Both envs (host `subsideo` and container `base`) end up package-identical; only the name differs. Task 7 ACCEPTANCE.md documents this invariant.
